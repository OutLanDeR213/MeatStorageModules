# DESIGN.md

## Чому не можна просто змінити `standard_price`

Умова задачі забороняє напряму міняти собівартість у `product.product` або `product.template`. Тому замість цього використовується `stock.valuation.layer` — саме так Odoo зберігає вартість кожної партії при FIFO.

Ми створюємо новий layer з `quantity=0` і `value=вартість_зберігання`. Це той самий механізм, яким Odoo реалізує Landed Costs — додаткові витрати на вже оприбутковані товари без фізичного руху запасів.

## Чому окрема модель для логу

`stock.valuation.layer` не зберігає нічого зайвого — тільки product, quantity, value. Нам же потрібно знати:
- до якої партії відноситься нарахування
- яка була початкова собівартість при надходженні
- який це за рахунком день зберігання

Тому є своя модель `stock.storage.cost` — вона і є журналом всіх нарахувань. На її основі будується звіт.

## Чому звіт через SQL view

`stock.storage.cost.report` — це `_auto = False`, тобто SQL view. Альтернатива — computed fields на основній моделі — була б повільнішою і складнішою для pivot. SQL дає просту агрегацію і добре працює з вбудованим pivot Odoo.

## Захист від подвійного запуску

На початку `_compute_daily_storage_costs` перевіряємо: якщо запис за сьогодні вже є — виходимо. Це захищає від ситуації коли cron запустився двічі (наприклад після рестарту сервера).

## Оптимізація запитів

Підрахунок кількості нарахувань для кожної партії (`adjustment_number`) спочатку робився окремим `search_count` в циклі — тобто N запитів на N партій. Зараз це один `read_group` до циклу, результат кешується в словнику.

Логи також створюються батчем через `self.create(logs)` — один INSERT замість N.

## Підтримка UoM

- Якщо товар в кг (або інших вагових одиницях) — конвертується в кг через `uom._compute_quantity()`
- Якщо в штуках — береться `product.weight`

## Ставка 0.01

Захардкоджена як константа класу `STORAGE_COST_RATE = 0.01`. При потребі можна винести в `res.config.settings` як параметр компанії.
